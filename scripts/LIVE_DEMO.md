# IntelliOps 现场演示 —— Postman + 页面操作指南

> **方式**：第 1 步用 Postman 模拟接收告警，后续 9 步全部在浏览器页面完成。
>
> **用时**：15-20 分钟（含讲解），适合现场一对一或大屏演示。

---

## 前置准备

### 1. 启动后端

```bash
cd d:/大模型/IntelliOps
pip install -r requirements.txt
# 可选：设置 LLM 环境变量（无 LLM 也能展示，自动降级规则模式）
# set LLM_PROVIDER=openai
# set OPENAI_API_KEY=sk-xxx
uvicorn src.backend.app:app --reload --host 0.0.0.0 --port 8000
```

### 2. 确认服务正常

浏览器访问 http://localhost:8000/health
返回类似：`{"status":"ok","skills":{"loaded":true,"count":9},"llm":{"configured":false}}`

### 3. 打开新 React UI

浏览器打开 http://localhost:8000/ui/

## 演示步骤（10 步）

---

### 第 1 步（Postman）：模拟告警接入

**Postman 操作：**

| 项目 | 值 |
|------|-----|
| Method | `POST` |
| URL | `http://localhost:8000/ingest/raw-alert` |
| Headers | `Content-Type: application/json` |
| Body | `{"severity":4,"source":"BPPM监控","content":"【SA-BPPM】MQ: MAR 31 04:15:44 BQREPXDEPS0 SEV=WAR QPS1 NamedQueue: QR.XMITQ1.QPS1.TO.QMA CURRENT DEPTH: 2197878 (CurrentDepth >= 2000000)。04:29 MQ Pageset使用率52.96%。04:37 Pageset使用率76.49%触发一级告警。2603批次投产后D+2日首次历史数据清理，单日清理25亿数据（QREP下刷5亿），远超QREP日常吞吐能力（~1亿/小时），导致MQ Pageset空间持续无法释放，数据下刷延迟无法在7:00前追平，影响DNF交易回切。9:56-9:58切换期间主机出现MaxTask，C-DBC/IPPS/RCPS-IBPS交易成功率分别降至98.19%/96.93%/90.5%。"}` |

**讲解（30 秒）：**
> "这是一条 BPPM 监控发来的原始告警。我们把 BPPM 监控的原始告警文本直接 POST 给系统——severity、source、content 三条原始信息。系统自动完成文本解析、系统匹配和变更关联，这就是"一条告警入，一个完整事件出"。现在到浏览器看结果。"

### 第 2 步（页面）：知识图谱拓扑上下文

**操作：** 页面左栏点击刚创建的事件 → 左栏"上下文"区自动展示

**页面会看到：**
- **受影响服务**：MQ、BOCS-DNF、BOCS-D
- **上下游依赖**（P0-1 新功能）：时间线顶部黄色警告条显示
  - 🠽 上游：BOCS-D（数据库）
  - 🠾 下游：C-DBC, IPPS, RCPS-IBPS
  - ⚠️ 上游变更：2603批次投产
- **关联变更**：chg-2603

**讲解要点（30 秒）：**
> "这是新升级的拓扑感知能力。页面顶部黄色条展示了完整的上下游依赖链：
> **上游**是 BOCS-D 数据库，**下游**是三个交易系统。更关键的是系统自动标注了**上游有变更**（2603 批次）—— 这是根因的高概率来源，排障时应该优先排查。"

### 第 3 步（页面）：异步根因诊断

**操作：** 点击 Header 中的 **"Copilot 诊断"** 按钮 → 异步提交 → 轮询等待 → 结果自动展示

**页面会看到（新增进度条）：**
- 诊断异步提交后，诊断面板显示进度条 + 当前步骤（如"正在执行技能路由..."）
- 完成后展示：
  - **综合置信度** + 渐变进度条
  - **活跃智能体**列表（每个 Skill 带图标 + 主 Skill 高亮）
  - **候选根因**（按置信度排序，带证据标签）
  - **日志分析摘要**
  - **建议下一步操作**

**讲解要点（1 分钟，核心卖点）：**
> "诊断现在是异步的 —— 点击后立刻返回，后台并行执行：技能路由、知识图谱分析、日志分析、案例检索。页面实时展示进度。
>
> 看诊断结果：
> ① **活跃智能体**：系统自动路由到 incident-diagnosis 和 log-analysis 两个 Skill
> ② **多步推理链**：现象 → 关联变更 → 容量对比 → 根因结论
> ③ **证据链**：每个根因都有可追溯的支撑证据
> ④ **拓扑感知**：如果上游系统有变更，诊断区顶部会高亮提示"

---

### 第 4 步（页面）：相似案例 & 知识资产

**操作：** 页面中下方 **KNOWLEDGE_BASE** 面板自动加载

**页面会看到：**
- **SIMILAR_CASES**（相似案例）：标题、匹配度百分比、根因摘要
- **KNOWLEDGE_ASSETS**（知识资产）：相关度百分比 + 可靠性标签
  - `RELIABLE`：多次验证有效
  - `DEGRADED`：误报率高
  - `UNVERIFIED`：尚未验证
- **HIGH_FREQ_PATTERNS**（高频模式）：聚合告警模式列表
- **触发聚合按钮**：点击后自动运行模式聚合 + Skill 更新

**讲解要点（30 秒）：**
> "下方知识库面板展示了三块内容：
> **相似案例** —— 自动向量检索匹配的历史复盘
> **知识资产** —— 每条知识都有可靠性标签（已验证/降级/未验证），系统会根据使用反馈动态调整权重
> **高频模式** —— 自动从历史故障中聚合高频场景
>
> 点击"RUN_AGGREGATION"按钮，系统会自动提炼模式并更新 Skill。"

---

### 第 5 步（页面）：拓扑感知脚本 + 执行闭环

**操作：** 点击 **LOAD_SCRIPTS** → 展示脚本列表 → 点 **DRY_RUN** 预执行 → 点 **EXECUTE** 执行

**页面会看到（新增 TOPO 标签）：**
- 脚本卡片按风险等级着色（低=绿、中=黄、高=红）
- **新增 TOPO 标签**（紫色）：标记为拓扑感知脚本
- **拓扑提示**（紫色斜体）：如"🠆 上游DB: BOCS-D"
- 脚本类型覆盖：
  - 🟢 采集关键错误日志（只读）
  - 🟢 检查上游数据库连接池与慢查询 ← **TOPO 感知**
  - 🟢 检查上游消息队列积压 ← **TOPO 感知**
  - 🟢 检查下游数据库影响 ← **TOPO 感知**
  - 🟡 检查服务核心指标（中度风险）
  - 🔴 重启受影响服务（高风险，需审批）

**讲解要点（45 秒）：**
> "这是新升级的拓扑感知脚本系统。每个脚本的推荐都基于知识图谱的上下文分析：
>
> 看带 **TOPO 标签** 的脚本 —— 系统知道受影响服务的上游有数据库，所以自动推荐了"检查上游数据库连接池与慢查询"的脚本；知道有 MQ 中间件，所以推荐了"检查消息队列积压"。
>
> 这是真正的智能化 —— 不是预设模板，而是根据当前事件的拓扑关系**动态生成**的处置建议。
>
> 闭环流程：DRY_RUN → EXECUTE → 结果自动反馈给 Copilot → 更新诊断置信度"

---

### 第 6 步（页面）：Copilot 多轮对话 + 讨论同步

**操作：** 在右下 **COMMS_CHANNEL** 输入框，依次发送：

| 轮次 | 发送方式 | 消息内容 |
|------|---------|---------|
| ① 开发讨论 | `SEND` (Ctrl+Enter) | `MQ通道堆积根因找到了吗？我们开发团队看日志发现QREP消费线程卡住了` |
| ② 确认根因 | `QUERY_AI` (Cmd+Enter) | `这个根因之前出现过吗？有没有类似案例？` |
| ③ 要处理步骤 | `QUERY_AI` (Cmd+Enter) | `现在该怎么处理？帮我推荐操作步骤` |

**页面会看到（新增功能）：**
- **讨论区双模式**：
  - `SEND` 按钮：发给团队（消息显示为 OPS 或 DEV 角色）
  - `QUERY_AI` 按钮：发给 Copilot（消息显示为 AI 角色）
- **P0-2 讨论证据提取**：AI 自动从讨论中提取关键证据
  - 发送①后，讨论区出现紫色 **"AI 从讨论中提取的证据"** 卡片
  - 显示：证据类型（🛡变更/✅确认/🔍发现）、提取摘要、可信度百分比
  - 同时自动写入**时间线**（如"协同·发现：开发者发现QREP消费线程卡住"）
- **P1-0 诊断更新**：Copilot 回答后，根因置信度自动调整

**讲解要点（1 分钟，核心卖点）：**
> "这是全新的协同智能功能。注意 **两个发送按钮**：
>
> **第一轮我用 SEND（发给团队）** —— 模拟开发同学在讨论中说"发现 QREP 消费线程卡住了"。看！系统自动识别出这是一条**关键证据**，紫色卡片显示"发现：开发者发现QREP消费线程卡住，可信度 82%"。同时时间线自动新增一条"协同·发现"记录。
>
> **第二轮我用 QUERY_AI（发给 Copilot）** —— Copilot 已经知道开发团队发现了什么，因为它自动同步了讨论中的证据。Copilot 回答时基于完整的上下文：诊断结果 + 开发反馈 + 相似案例。
>
> **第三轮继续问操作步骤** —— Copilot 知道我们已经排查过什么、发现了什么，给出针对性的处置建议"

---

### 第 7 步（页面）：排障状态机

**操作：** 打开浏览器开发者工具或 Postman 查看

> 新增 **P1-1 排障状态机**。目前页面在诊断面板中展示 **excluded 和 verified** 列表。完整四象限（to_verify / high_risk / verified / excluded）可通过 API 查看：
> `GET http://localhost:8000/incident/{inc_id}/investigation-state`

**页面会看到（诊断面板底部）：**
- 🔴 已排除 3 项：列出已排除的排查方向
- 🟢 已验证 2 项：列出已确认的排查结论
- 高阶版本：四象限可拖拽操作

**讲解要点（20 秒）：**
> "诊断面板底部展示了排障状态机。已排除的排查方向不会在下次 Copilot 对话中被重复推荐，避免反复排查相同方向浪费时间的痛点。"

---

### 第 8 步（页面）：生成复盘报告

**操作：** 点击 Header 中的 **"生成复盘"** 按钮 → 弹出复盘弹窗

**页面会看到：**
- 弹窗展示：
  - 报告 ID + 状态
  - **根因结论** + 置信度
  - **改进建议**列表（带优先级）
  - **时间线快照**（从告警到恢复的完整事件流，包含讨论同步证据）
- 关闭弹窗后，事件状态自动更新为 Resolved

**讲解要点（1 分钟）：**
> "点击生成复盘，系统自动生成完整报告。注意改进建议中包含从讨论中提取的内容 —— 因为 **讨论同步代理** 记录了开发团队发现 QREP 消费线程问题的过程，这些信息也进入了复盘报告。
>
> 一次故障，三重沉淀：
> ① 一份完整的复盘报告
> ② 一批可执行的改进任务
> ③ 知识蒸馏 → 规则 + 模式 → Skill 自动更新"

---

### 第 9 步（页面）：查看知识资产与高频模式聚合

**操作：** 刷新页面 → 查看 **KNOWLEDGE_BASE** 面板

**页面会看到：**
- 知识资产卡片带**可靠性标签**（RELIABLE / DEGRADED / UNVERIFIED）
- 每条资产显示：验证次数 / 误报次数 / 动态权重
- **高频模式列表**：聚合后的告警模式
- **Skill 更新日志**：历史更新记录
- 点击 **RUN_AGGREGATION** 按钮触发聚合

**讲解要点（20 秒）：**
> "现在知识资产每条都有**可靠性标签** —— 系统会根据每次使用后的反馈动态调整。验证多的资产权重高，误报多的降级。这是从'知识积累'到'知识可信度管理'的升级。"

---

### 第 10 步（页面）：切换场景 + 角色权限 + 全景

**① 切换到 CPU 风暴场景（Postman）：**
```json
{"scenario": "cpu_spike", "severity": 4}
```
页面刷新后点击新事件，重复步骤 2-8。

**② 查看用户角色权限：**
- 左上角用户选择：`ui-user`（运维） vs `ops-viewer`（值班查看）
- 切换为 `ops-viewer` 后，脚本面板的 EXECUTE 按钮变为 **LOCKED**
- 体现 RBAC 权限控制

**内置 6 个场景：**
| Scenario | 故障描述 | TOPO 感知重点 |
|----------|---------|--------------|
| bocs_dnf_sync_delay | 2603 投产 DNF 数据同步延迟 | 变更→MQ→数据同步链 |
| db_timeout | BOCS-D 数据库连接池耗尽 | 数据库上游检查 |
| cpu_spike | 主机 CPU 99% GC 风暴 | 主机→微服务→下游交易 |
| mq_backlog | QREP 通道堆积 12 万条 | MQ→消费端延迟 |
| network_flap | MCIS 接口超时率 18% | 网络→分行→第三方链 |
| mcis_branch_thirdparty | 湖南分行网银成功率 45.5% | 前置→分行→第三方加密 |

## 常见问题

**Q：讨论区 SEND 和 QUERY_AI 有什么区别？**
A：`SEND` 发给团队（OPS/DEV 角色），记录到讨论区，`DiscussionSyncAgent` 会自动提取证据；`QUERY_AI` 发给 Copilot，会调用 LLM 分析并基于完整上下文（含讨论证据）回答。

**Q：TOPO 标签的脚本从哪里来？**
A：系统根据 KG 上下文中受影响服务的上下游依赖类型动态生成。上游有数据库 → 生成数据库检查脚本；有 MQ → 生成 MQ 检查脚本。不需人工预设。

**Q：讨论同步代理能提取什么类型的内容？**
A：支持 8 种证据类型：变更（🛡）、确认（✅）、操作执行（⚡）、恢复（🟢）、持续（🔶）、发现（🔍）、交接（🔧）。每条证据有可信度评分。

**Q：知识资产的可靠性标签如何更新？**
A：每次脚本执行或诊断后，用户可以对知识资产进行反馈（验证通过/误报）。系统使用加权统计动态调整 —— 验证多则权重上升，误报多则降级为 DEGRADED。
