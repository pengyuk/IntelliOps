# 故障应急排查模块架构（原型研发）

**项目定位**：短周期原型研发，目标验证大模型驱动的故障认知与协同能力；两人并行迭代，通过明确接口（JSON Schema）解耦。

## 1. 总体架构概览

原型建议采用“数据层 + 知识层 + 能力层 + 应用层”的分层设计：

- 数据层：多源输入、标准化清洗、变更与工单场景化数据
- 知识层：知识库（文档/案例+标签+向量检索），按需演进为本体/知识图谱
- 能力层：Skill 引擎、自动化 Harness、协同引擎
- 应用层：事故看板、排查助手、复盘闭环、运维/开发协同门户

```
+----------------+    +------------------+    +---------------------+    +------------------------+
| 数据层         | -> | 知识层           | -> | 能力层              | -> | 应用层                |
| - T-CMDB       |    | - 知识库       |    | - Skill             |    | - 事故看板            |
| - T-OPM        |    | - 文档/案例/标签  |    | - Harness           |    | - 现场排查助手        |
| - 优云导出     |    | - 故障案例库     |    | - 协同引擎          |    | - 复盘与任务管理      |
| - 变更/工单    |    | - SOP 模板       |    | - 规则引擎          |    | - 知识库与搜索        |
+----------------+    +------------------+    +---------------------+    +------------------------+
```

## 2. 层级与模块说明

### 2.1 数据层

目标：打通并结构化故障排查所需的多源数据。

主要模块：

- 数据采集
  - 支持 T-CMDB、T-OPM、优云系统导出、变更清单、工单、容量数据
- 数据标准化
  - 统一实体命名、时间序列、关联键
- 数据存储
  - 对象存储/关系型存储/图数据库

输出：标准化事件、资产、变更、告警、容量、工单记录

### 2.2 知识层

目标：先用知识库（文档/案例 + 标签/元数据 + 向量检索）快速实现可用的检索与推荐能力，后续按需演进为本体/知识图谱以支持更强的多跳查询与推理。

主要模块（初期优先级）：

- 知识库
  - 文档/复盘文本、SOP、操作模板、工单摘要、变更要点，配合标签与元数据
- 向量检索/文本索引
  - 基于 embeddings 或 TF-IDF 的相似案例检索
- 故障案例库
  - 历史故障事件、排查路径、根因结论、处置命令（作为知识库的主要内容）
- SOP/处置模板库
  - 标准化应急操作步骤与自动化脚本映射

输出（初期）：可检索的运维知识集、相似案例召回与推荐步骤

演进方向：在数据质量和使用场景验证后，逐步补充结构化实体与关系，形成子图入图，再迁移或并行构建本体/KG。

### 2.3 能力层

目标：构建支持排查决策、自动化执行与协同联动的能力。

主要模块：

- Skill 引擎
  - 面向应急场景的问答与排查指引
  - 通过知识层（知识库/子图）提供“故障问题→建议步骤”能力
- Harness 自动化框架
  - 应急操作模板执行、脚本管理、校验与回滚机制
- 协同引擎
  - 角色通知、事件订阅、现场信息共享、任务流转
- 规则/推荐引擎
  - 根据故障特征自动推荐排查路径、关联变更、推送负责人

输出：智能故障分析、自动化执行、信息协同

### 2.4 应用层

目标：将能力模块落地成面向用户的运维/开发协同工具。

主要模块：

- 事故看板
  - 全局态势、故障影响范围、实时进度、关键指标
- 现场排查助手
  - 结构化故障信息卡片、日志入口、快速诊断路径
- 自动化处置库
  - 可执行的应急指令模板与脚本
- 协同通信与联动
  - 角色协同、消息通知、会议/通报记录
- 复盘与闭环管理
  - 复盘事件、改进任务、落实追踪、知识库更新
- 知识库与经验搜寻
  - 历史故障检索、故障标签、SOP查询

输出：可操作的运维平台与故障应急工作空间

## 3. 核心能力映射

| 需求痛点 | 解决方案 | 对应模块 |
|---|---|---|
| 个人经验依赖 | 知识库+案例检索，标准化片段与推荐步骤 | 知识层、Skill 引擎 |
| 自动化能力不足 | Harness 自动化执行与模板化操作 | 能力层、应用层 |
| 日志/变更查询分散 | 数据层多源采集、统一查询入口 | 数据层、应用层 |
| 协同联动弱 | 协同引擎+角色通知机制 | 能力层、应用层 |
| 复盘闭环缺失 | 复盘任务管理、知识库更新机制 | 应用层、知识层 |

## 4. 推荐模块架构

### a. 知识层（知识库，演进为本体/KG）

首选采用知识库以快速交付检索与推荐能力：

- 内容：复盘文本、SOP、操作模板、工单/变更摘要、标签与元数据
- 能力：文档检索、相似案例召回、模板化步骤推荐
- 目标：通过实际使用积累常见查询模式与标签，为后续本体/KG 建模提供数据驱动输入

### b. 可选：知识图谱模块（KG，后期演进）

当需求明确且数据质量成熟时，可按阶段引入 KG：

- 作用：将多源数据按照本体构建图结构以支持复杂的多跳查询与推理
- 场景：跨服务根因追溯、路径可解释的推荐、长期语义化知识沉淀

### c. Skill 模块

负责对运维/开发提供智能交互接口：

- 问题识别：故障描述、影响范围、历史类比
- 结果输出：推荐诊断步骤、执行建议、相关文档
- 场景指引：现场信息补录、手工操作提示、自动化命令推荐

### d. Harness 模块

负责自动化执行与闭环支撑：

- 应急操作模板管理
- 执行权限与日志记录
- 变更/工单自动打通
- 复盘任务分配与跟踪

## 5. 典型数据流

1. 故障触发
2. 数据层抓取告警、工单、变更、CMDB依赖
3. 知识层构建知识库/关联视图，关联责任人、历史案例、SOP
4. Skill 引擎生成排查建议，推荐下一个动作
5. Harness 执行自动化操作或生成工单
6. 应用层展示事故看板与现场卡片，驱动协同沟通
7. 事故结束后，复盘任务入库，更新知识库/子图与经验库

## 6. 原型阶段建议

- 优先构建“事故看板 + 现场排查助手 + 知识库推荐（文档+向量检索）” MVP
- 以“1~2个产品标准化数据集”为样例输入
- 后续补充“自动化处置 + 复盘闭环 + Skill 交互”
- 将“知识库 → 子图入图 → 本体/KG”作为迭代能力框架

### 并行分工（两人·能力轴·低耦合）

- `Person A：大模型认知引擎`
  - 负责：LLM 集成、Prompt 工程、Root Cause Reasoner、Log Analyzer、Credibility Framework、Knowledge Distiller
  - 输出：标准化推理结果（JSON）、置信度、依据链、API 文档

- `Person B：数据平台与应用层`
  - 负责：ETL、知识库、Incident Graph、Investigation State Machine、War Room Agent、UI 与执行层
  - 输出：标准化数据接口（JSON Schema）、向量索引、事件展示与执行反馈

解耦要点：双方在阶段 A 结束前完成接口 Schema 对齐（REST/JSON），各自可独立迭代，集成点最小化。

## 7. 痛点映射与架构设计要点

- 痛点：知识/经验不足 → 方案：`Ontology + KG + 案例库` 提供结构化知识与检索
- 痛点：手工处置不可审计 → 方案：`Harness` 强制模板化、审批与审计 ID
- 痛点：信息分散导致沟通成本高 → 方案：`事故看板 + 现场卡片 + 协同引擎` 实时共享
- 痛点：复盘不闭环 → 方案：复盘模块与任务流转、自动催办与 KPI 看板

## 8. 关键接口与数据契约示例

建议采用 JSON REST 接口进行模块解耦，示例：

- `POST /ingest/alerts` 接收告警：

  {
    "alert_id": "al-1",
    "severity": 4,
    "metric": "payment.latency",
    "timestamp": "2026-06-01T10:00:00Z",
    "source": "T-OPM",
    "context": {"service_id": "svc-001", "host": "host-01"}
  }

- `GET /incident/{id}` 返回事件详情（含 KG 关联节点与推荐步骤）

  {
    "incident_id": "inc-1",
    "status": "Investigating",
    "summary": "支付延迟",
    "kg_context": {"services": [...], "recent_changes": [...]},
    "recommendations": [{"step":"收集日志","action_id":"act-001"}]
  }

- `POST /action/execute` 执行 Harness 模板（返回 audit_id）

  {
    "action_id": "act-001",
    "params": {"since":"30m"},
    "requested_by": "user-1"
  }

返回示例： `{ "audit_id": "exec-1234", "status": "accepted" }`

## 9. 知识图谱（KG）设计细化

- 实体优先级：`Incident`、`Service`、`Host`、`Alert`、`Change`、`WorkOrder`、`Action`、`Person`、`SOP`。
- 每个实体至少包含统一 ID 与时间戳、来源字段（source_system）以便追溯。
- 边类型示例：`triggers`, `related_to`, `runs_on`, `affected_by`, `resolved_by`, `documented_in`。
- 相似故障检索：使用标签 + 特征向量（简单 TF-IDF 或 embedding）做候选，再通过图路径相似度排序。

## 9.1 是否引入知识图谱（KG）——决策与分阶段路线

结论要点：KG 能带来长期的多跳关联与推理能力，但会增加建模与运维成本。建议采取渐进式策略，根据数据质量与业务需求决定是否以及何时扩展为完整 KG。

分阶段建议：

- 阶段 0（快速验证，MVP）：不建完整 KG。统一实体 ID 及字段，把数据存为结构化 JSON/关系表（SQLite/Postgres），并用文本/向量检索做历史故障匹配与推荐。
- 阶段 1（子图入图）：定义最小本体（Incident/Service/Host/Change/Alert），把关键服务与最近 3–6 个月的变更/告警入图，验证多跳查询的收益。
- 阶段 2（全量 KG）：在验证出明显收益后，迁移到图数据库（Neo4j/JanusGraph），并实现规则推理、图搜索及更复杂的根因分析能力。

替代和补充技术：

- 文档 + 向量检索：对历史复盘、日志片段做 embeddings，以快速获得类似案例。
- 事件关联规则引擎：用时间窗口、变更匹配等规则做实时关联（适合低成本实现）。

何时优先引入 KG 的信号：

- 需要频繁做多跳依赖追溯（例如：服务→服务→主机→变更），单表/SQL 查询变得复杂且性能不足；
- 推荐与根因推理需要解释性强的路径证据；
- 希望把长期的知识与实践（SOP、案例）以可查询语义网络形式沉淀并复用。

工程建议：

- 在 MVP 阶段确保实体建模一致（统一 ID、时间戳、source_system），为未来入图保留字段；
- 在阶段 1 实现可回滚的子图同步脚本与质量校验，避免脏数据污染图；
- 为阶段 2 规划图 DB 的扩展（索引策略、缓存、查询限制、异步推理任务）。

## 10. 安全、合规与审计

- 所有自动化执行必须产生审计记录（exec_id、请求人、时间、输入、输出、审批链）。
- 日志/监控数据需支持脱敏与最小化访问：仅对有权限的角色开放明文日志。
- 对外接口应支持 RBAC 与 API Token；敏感操作需二次确认或审批流。

## 11. 可观测性与运维

- 监控项：API 延迟、ETL 失败率、KG 入图延迟、Action 执行成功率、复盘任务逾期率。
- 日志：操作日志、审计日志、系统日志应分类存储并建立索引以便追溯。

## 12. 性能与扩展性考虑

- KG 查询：低延迟场景建议在内存缓存热点子图；复杂推理放到异步批任务。
- ETL：批处理 + 流处理结合（Kafka + 调度器），避免在高峰期阻塞查询路径。

## 13. 测试、CI/CD 与部署建议

- 单元测试：各服务逻辑、入图映射规则、Skill 推荐规则。
- 集成测试：模拟告警流、执行 Action、复盘任务闭环。
- 部署：容器化（Docker）+ k8s（小规模 dev 集群），分环境（dev/stage/prod）。

## 14. 示例序列（故障触发到复盘）

1. 告警从 T-OPM 到 `POST /ingest/alerts`。
2. 平台自动创建 `Incident` 并在看板上提醒值班运维。
3. 值班在 `事件详情` 填写现场卡片，Skill 返回推荐步骤并列出相关 `Change`。
4. 选择 `Action` 执行（若为敏感操作，触发审批），执行结果回写事件与知识库/子图。
5. 故障恢复后，进入 `Postmortem`，生成改进任务并分配责任人。

## 15. Glossary（术语）

- KG：Knowledge Graph，知识图谱。
- SOP：Standard Operating Procedure，标准操作流程。
- Harness：自动化执行框架。
- Skill：面向故障排查的交互/能力插件。
