# 故障应急排查平台升级方案

## 核心认知：从"智能运维平台"升级到"大模型驱动的应急认知系统"

基于对当前方案的深度分析，本文档重新定位项目方向，突出**大模型在故障应急中的核心价值**。

---

## 一、当前方案的层级分析

### 第一层：传统平台能力（已覆盖）

✅ 知识库（文档/案例 + 标签/元数据）  
✅ 相似案例召回（向量检索）  
✅ SOP 推荐与自动化脚本  
✅ 事故看板与事件管理  
✅ Action 执行与审计  
✅ 基础流程协同  

**本质**：信息聚合、经验复用、自动化编排 → 必要但不充分

### 第二层：大模型能力缺口（急需补充）

❌ 故障推理链（现象 → 系统 → 依赖 → 变更 → 根因 → 排查优先级）  
❌ 日志认知能力（自动摘要、异常检测、多源关联）  
❌ 动态问答式排查（多轮对话、状态记忆、实时建议）  
❌ 多角色协同智能体（自动同步、状态广播、角色通知）  
❌ 现场排查状态机（已验证/待验证/高风险跟踪）  
❌ 故障知识蒸馏（自动从事故提炼根因规则、预警、SOP、脚本）  
❌ 经验传播机制（故障演练模式、AI 评分、新人培养）  
❌ 可信度与可解释性设计（推荐置信度、依据链、风险提示）  

**本质**：推理型能力、协同智能体、闭环学习 → 真正的大模型价值所在

## 二、重新设计的核心能力

### 缺口 1：Root Cause Reasoning Engine（根因推理引擎）

**当前状态**：
```
告警 → 历史案例检索 → 推荐步骤
```

**升级后**：
```
告警 + CMDB + 拓扑 + 变更 + 日志摘要 + 历史案例
  ↓
Incident Graph（事件图）
  ↓
LLM 推理
  ├─ 哪些系统可能有问题
  ├─ 哪些依赖链异常
  ├─ 哪些变更最可疑
  ├─ 为什么优先查这个
  └─ 下一步最优动作
  ↓
排序 + 可信度 + 依据链
```

**输出示例**：
```json
{
  "root_causes": [
    {
      "cause": "数据库连接池参数与国产DB不兼容",
      "confidence": 0.82,
      "evidence": [
        "最近变更：连接超时参数降低",
        "3个历史相似案例",
        "当前连接等待时间>5s"
      ]
    }
  ],
  "reasoning_chain": "应用异常 → 网关 502 → DB 连接等待激增 → 连接池配置不匹配国产数据库特性",
  "next_actions": [
    {
      "action": "检查数据库连接池日志",
      "priority": 1,
      "rationale": "最直接定位根因"
    }
  ]
}
```

### 缺口 2：Intelligent Log Analysis（日志智能分析）

**当前状态**：
```
"日志依赖运维查询"
```

**升级后**：

1. **日志自动摘要**
```
过去 15 分钟关键事件：
- 数据库连接超时激增（从 0.1% → 5%）
- 网关 502 从 0 → 200/min
- 某节点 GC 频率异常（从 10s 一次 → 1s 一次）
```

2. **异常模式识别**
```
这个报错"XXX" 过去从未出现
与上次 2026-05-15 故障 99% 相似
```

3. **多源日志关联**
```
应用日志异常 + 中间件日志 + 主机日志 + K8s Event
 → 联合推理生成统一时间线
```

### 缺口 3：Incident Copilot（事故副驾）

**升级 Skill 引擎**：

从固定问答 → 动态多轮对话 + 状态跟踪 + 自动证据收集

**交互示例**：
```
运维：故障现象是 CPU 高但没流量，怎么办？
AI：检测到最近 2 小时有变更，是否相关？
运维：有一个连接池参数修改
AI：建议立即检查 DB 连接泄漏，已生成命令
    [执行命令按钮] [查看日志] [相似案例]
运维：连接数 2000 但应该最多 100
AI：确认连接泄漏！根因很可能是参数设置
    立即建议：
    1. 执行连接重置脚本（已审批可执行）
    2. 同时收集堆栈信息用于后续根因分析
```

**能力**：
- 多轮上下文记忆
- 排障状态跟踪
- 动态提问式诊断
- 自动生成排查路径
- 自动收集证据与时间线

### 缺口 4：War Room Agent（战情室智能体）

**自动同步现场状态**：

```
19:32 数据库连接池异常（检出）
      ↓ AI 行动
      自动 @ DBA, DB 变更负责人
      
19:35 已执行连接重建（运维操作）
      ↓ AI 行动
      更新状态：已缓解（待验证）
      生成时间线
      通知应用团队
      
19:40 网关恢复 50%（自动检测）
      ↓ AI 行动
      自动识别最近变更来自支付团队
      提示：可能影响其他依赖方
      
19:45 完全恢复
      ↓ AI 行动
      自动生成会议纪要
      触发复盘流程
```

**核心能力**：
- 实时状态同步
- 自动生成时间线
- 角色自动通知
- 统一现场视图（运维/开发/DBA 各自视角聚合）
- 自动生成会议纪要

### 缺口 5：Investigation State Machine（排查状态跟踪）

**维护现场排查状态**：

```json
{
  "incident_id": "inc-2026-001",
  "verified": {
    "网络连通性": {"status": "正常", "checked_by": "runop-ai", "time": "19:33"},
    "Redis": {"status": "正常", "checked_by": "runop-ai", "time": "19:34"}
  },
  "to_verify": {
    "DB连接泄漏": {"priority": "high", "suggested_action": "act-001"},
    "应用内存泄漏": {"priority": "medium"}
  },
  "high_risk": {
    "刚发生配置变更": {"change_id": "chg-100", "risk": "高"}
  },
  "excluded": {
    "磁盘空间": "已排除，磁盘使用率 40%"
  }
}
```

**用处**：新加入的人能立即接管，不用再问"查到哪了"。

### 缺口 6：Fault Knowledge Distillation（故障知识蒸馏）

**从复盘自动蒸馏系统能力**：

复盘后，AI 从以下来源自动提炼：
- 群聊记录
- 工单描述
- 操作日志
- 时间线
- 日志片段

**自动生成**：

1. **根因规则**
```
规则：连接池等待时间 > 5s 且变更涉及连接参数
→ 触发告警：连接泄漏高风险
```

2. **预警规则**
```
新增监控：
- 连接池等待时间分布
- GC 频率异常
- 特定错误码激增
```

3. **标准 SOP**
```
新增 SOP：处理国产数据库连接异常
步骤：
1. 检查连接池配置
2. 执行连接重置脚本
3. 验证 DB 日志
```

4. **自动化脚本**
```
识别高频操作：
"连接重置" - 重复率 72%
建议：沉淀为自动 Action
```

### 缺口 7：Incident Simulator（故障演练）

**AI 驱动的新人培养**：

```
模式：演练
1. AI 模拟真实故障
   - 生成虚拟现象
   - 注入虚拟日志
   - 模拟告警
   
2. 新人进行排查
   - 提问与操作
   - AI 实时指导与评分
   
3. AI 评分与反馈
   - 诊断准确性
   - 效率评分
   - 改进建议
```

**价值**：大幅加快新人对信创系统故障的认知。

### 缺口 8：Credibility & Explainability（可信度与可解释性）

**高风险场景必须**：

1. **可信度机制**
```json
{
  "recommendation": "重启数据库",
  "confidence": 0.72,
  "evidence": [
    "相似案例 3 个（可信度贡献 0.5）",
    "最近变更 1 个相关（贡献 0.2）",
    "日志异常确认（贡献 0.02）"
  ]
}
```

2. **可解释性**
```
为什么推荐这个动作：
- 根因链：应用异常 → 连接失败 → 连接池耗尽
- 影响链：连接耗尽 → 新请求无法获得连接 → 服务不可用
- 修复链：增加连接 / 重置泄漏 → 恢复可用
```

3. **人工确认门禁**
```
重启数据库
⚠ 风险：3-5 分钟业务中断
审批状态：待批准（需 DBA 确认）
建议先：验证连接泄漏后再执行
```

4. **Action 沙箱**
```
执行前：
- Dry Run 预演
- 影响范围分析
- 回滚计划验证

执行中：
- 实时进度
- 异常自动回滚

执行后：
- 验证效果
- 审计日志
```

## 四、分阶段实施路线

### 阶段 A（快速 MVP，保留）
- 知识库 + 相似案例检索
- 事故看板 + 基础 Action 执行
- **预期**：降低 MTTR 15-20%

### 阶段 B（大模型认知能力，新增）
- Root Cause Reasoning Engine
- Intelligent Log Analysis
- Incident Copilot（升级 Skill 引擎）
- Investigation State Machine
- **预期**：降低 MTTR 30-40%，提升诊断准确性

### 阶段 C（协同与闭环，新增）
- War Room Agent
- Fault Knowledge Distillation
- Incident Simulator
- 自动复盘与 SOP 生成
- **预期**：复盘闭环率 80%+，新人培养时间缩短 60%

## 六、最终定位

**从**：智能运维平台（知识库 + 流程 + 自动化）

**升级为**：大模型驱动的应急认知系统

**核心价值**：
- **推理型**：能解释"为什么" 而不仅仅"怎么做"
- **协同型**：多角色智能体自动同步协作
- **学习型**：自动从故障中蒸馏经验与预警规则
- **可信型**：每个建议都有可信度、依据、风险评估
