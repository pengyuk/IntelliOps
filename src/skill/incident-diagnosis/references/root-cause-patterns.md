# 常见根因模式参考

## 延迟类故障

### 模式 1：数据库连接池耗尽
- **现象**：P99 延迟突然升高，但 QPS 未增长
- **证据链**：应用日志 `connection pool exhausted` → 数据库 `active connections` 接近 max → 慢查询增多
- **排查路径**：
  1. 检查连接池配置（max_connections、timeout）
  2. 查看最近是否有连接池参数变更
  3. 分析慢查询日志，找阻塞查询
- **置信度**：有连接池日志时 0.80+，仅凭延迟推测时 0.55

### 模式 2：下游依赖超时
- **现象**：特定接口延迟升高，其他接口正常
- **证据链**：应用日志 `timeout calling <service>` → 下游服务健康检查失败 → 网络延迟或负载高
- **排查路径**：
  1. 确认下游服务状态（健康检查、资源使用率）
  2. 检查网络链路（ping、traceroute）
  3. 查看下游是否有级联故障
- **置信度**：有时序调用链数据时 0.85+，仅应用日志时 0.60

### 模式 3：GC 暂停
- **现象**：间歇性延迟尖刺，与 GC 日志时间吻合
- **证据链**：JVM GC 日志 `GC pause exceeded` → 堆内存使用率 > 80% → 对象分配速率异常
- **排查路径**：
  1. 检查堆内存配置（-Xmx、-Xms）
  2. 分析近期是否有内存泄漏
  3. 查看是否有大对象频繁创建
- **置信度**：有 GC 日志时 0.75+，仅推测时 0.40

## 错误率类故障

### 模式 4：配置变更引入
- **现象**：部署/变更后错误率突然上升
- **证据链**：变更记录时间与故障窗口高度重合 → 错误日志 `ClassNotFoundException` / `configuration key changed` → 回滚后恢复
- **排查路径**：
  1. 获取故障窗口内的所有变更记录
  2. 对比变更前后的配置差异
  3. 优先回滚最近变更，观察是否恢复
- **置信度**：变更窗口匹配 + 回滚验证通过时 0.90+

### 模式 5：资源耗尽（OOM/CPU）
- **现象**：服务间歇性不可用，伴随 OOMKilled 或 CPU Throttling
- **证据链**：K8s Events `OOMKilled` → 内存使用曲线陡升 → 容器 limit 不匹配
- **排查路径**：
  1. 检查容器资源限制配置
  2. 分析内存/CPU 使用趋势
  3. 检查是否有流量突增
- **置信度**：有 K8s Events + 资源曲线时 0.85+

## 分类速查表

| 关键词 | 最可能模式 | 优先检查 |
|--------|-----------|---------|
| 延迟、慢、timeout | 模式1 连接池 | 连接池配置 |
| 特定接口慢 | 模式2 下游超时 | 下游健康状态 |
| 间歇性卡顿 | 模式3 GC暂停 | JVM 堆配置 |
| 部署后出错 | 模式4 配置变更 | 变更记录 |
| OOM、重启 | 模式5 资源耗尽 | 容器 limits |
