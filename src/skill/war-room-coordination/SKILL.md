# War Room 协同

## 何时使用

- 故障涉及多个团队需要同步信息
- 需要通知特定角色（DBA、开发、业务方）
- 需要广播排查进度
- 需要指派任务给特定人员

## 角色与通知规则

参考 [角色矩阵](./references/role-matrix.md) 确定通知对象：

| 故障类型 | 必须通知 | 建议通知 |
|---------|---------|---------|
| 数据库相关 | DBA、相关服务运维 | 应用开发 |
| 网络相关 | 网络运维 | 受影响服务负责人 |
| 应用错误 | 应用开发、运维 | 产品经理 |
| 容量/资源 | 基础设施运维 | 应用开发、DBA |
| 安全事件 | 安全团队 | CTO、法务 |

## 核心流程

### 步骤 1：获取可用角色

查询系统注册用户：

```
GET {{base_url}}/auth/users
```

返回用户列表及其角色：
- `operator`：运维人员（可执行操作）
- `developer`：开发人员（可查看、评论）
- `approver`：审批经理（可审批高风险操作）
- `admin`：系统管理员

### 步骤 2：发送协同消息

调用讨论区 API：

```
POST {{base_url}}/incident/{{incident_id}}/discussion
Content-Type: application/json

{
  "author": "{{current_user_id}}",
  "message": "@developer 已确认连接池耗尽，建议检查最近的连接池配置变更。当前排查进度：已完成日志分析和指标检查。",
  "message_type": "maintenance",
  "mentions": ["dev-user"]
}
```

`message_type` 说明：
- `maintenance`：运维侧消息
- `development`：开发侧消息
- `decision`：关键决策（会自动写入时间线）
- `conclusion`：阶段性结论
- `handoff`：任务交接

### 步骤 3：同步排查进度

定期或在关键节点同步进度：

```
📢 进度同步 (#inc-1)

✅ 已完成：
  • 日志分析 — 发现连接池耗尽 + 慢查询
  • 指标检查 — 活跃连接 450/500

🔄 进行中：
  • 变更记录核对 — @developer 确认中

⏳ 待执行：
  • 连接池配置验证
  • 慢查询优化评估
```

### 步骤 4：通知状态变更

当故障状态变化时，自动通知相关人员：

- 故障升级（Investigating）：通知开发负责人和业务联系人
- 需审批：通知审批经理
- 恢复确认：通知所有参与者
- 复盘完成：通知相关方审阅

### 步骤 5：生成协同摘要

故障结束后生成协同摘要：

```
📋 协同摘要

参与人员：运维人员、开发人员、审批经理
讨论消息：12 条
关键决策：
  1. 临时放宽连接池上限（运维提议，开发确认）
  2. 长期方案：添加连接池监控告警（开发负责）

时间线：
  14:20 告警触发
  14:25 运维开始排查
  14:30 @developer 请求确认配置
  14:35 开发确认最近变更
  14:40 审批通过，执行修复
  14:50 故障恢复
```

## 注意事项

- 避免在讨论区发送敏感信息（密码、密钥）
- 关键决策（message_type: decision）会永久记录在时间线
- 如果 WebSocket 已实现，消息会实时推送到所有在线用户
