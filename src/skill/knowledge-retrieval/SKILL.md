# 知识检索

## 何时使用

- 需要查找与当前故障相似的历史案例
- 需要了解某个服务的依赖关系（上游/下游）
- 需要获取标准操作流程（SOP）
- 需要查询知识图谱中的实体和关系

## 核心流程

### 步骤 1：确定查询类型

根据用户意图选择查询方式：

| 用户意图 | 查询类型 | API |
|---------|---------|-----|
| "有没有类似故障" | 相似案例检索 | `GET /incident/{id}/related-cases` |
| "支付服务依赖哪些服务" | 子图查询 | `GET /kg/subgraph?node_id=svc-001&depth=2` |
| "查一下支付相关" | 关键词搜索 | `GET /kg/query?q=支付` |
| "svc-001 的历史故障" | 服务历史 | `GET /kg/history?service_id=svc-001` |
| "这个故障影响了什么" | 事故 KG 图 | `GET /kg/incident/{id}` |

### 步骤 2：执行查询

根据类型调用对应的 API。所有 API 都是 GET 请求，可直接在浏览器或 curl 中测试。

### 步骤 3：格式化结果

**相似案例结果**：

```
📚 相似历史案例

1. inc-old-1：支付网关超时（匹配度：5）
   根因：数据库连接池配置不当
   处置：扩大连接池 + 索引优化
   可用脚本：check_pool.sh

2. inc-old-2：订单服务延迟（匹配度：3）
   根因：下游库存服务响应慢
   处置：临时降级 + 扩容库存服务
```

`similarity_score`：1-2=低相似，3-4=中相似，5+=高相似（可直接复用处置方案）。

**KG 子图结果**：

```
🔗 服务依赖关系：支付网关（svc-001）

上游依赖方：
  • 订单服务（svc-002）— 调用支付网关
  • 用户服务（svc-003）— 调用支付网关

下游依赖：
  • 数据库（db-postgres-01）— 被支付网关依赖
  • Redis 缓存（cache-redis-01）— 被支付网关依赖

关联变更：chg-100（连接池参数调整）
关联告警：al-1（支付延迟告警）
```

### 步骤 4：注入诊断上下文

检索结果可注入当前诊断。如果存在活跃的诊断会话，通过 `POST /copilot/chat` 发送：

```
"知识检索结果：发现 2 个高相似度案例，根因均为连接池配置问题。当前故障的服务依赖链包括数据库和Redis。"
```

Copilot 会根据检索结果更新根因假设和推荐步骤。

## 注意事项

- KG 数据来自 `sample_kg.json`（原型阶段），生产环境需接入真实图数据库
- 相似案例基于简单集合交集计分，后续升级为向量检索
- 检索结果仅供参考，最终决策需结合当前故障的实际上下文
