# 应急脚本代码审查

> 改编自通用 Code Review Skill，针对运维脚本的安全性和正确性定制。

## 何时使用

- AI 生成新脚本后需要人工审查
- 执行 `risk_level: high` 的脚本前必须审查
- 脚本标记为 `permanent` 入库前需要质量把关
- 怀疑现有脚本有安全风险时

## 审查清单

### 安全审查（必检）

| 检查项 | 说明 | 风险 |
|--------|------|------|
| 硬编码凭证 | 脚本中是否有密码、API Key、Token | 🔴 严重 |
| 权限提升 | 脚本是否使用 `sudo`、`root` | 🔴 需确认必要性 |
| 写操作 | 脚本是否有 `rm`、`DROP`、`DELETE`、`ALTER` | 🔴 需审批 |
| 网络调用 | 脚本是否向外部发送数据 | 🟡 需确认目的 |
| 注入风险 | 脚本是否拼接用户输入到命令中 | 🟡 需参数化 |
| 文件操作 | 脚本是否修改配置文件、系统文件 | 🟡 需备份 |

### 正确性审查

| 检查项 | 说明 |
|--------|------|
| 错误处理 | 是否有 `set -e` / `try-except` / 退出码检查 |
| 超时控制 | 网络/数据库操作是否有超时设置 |
| 幂等性 | 重复执行是否安全（尤其是写操作） |
| 回滚方案 | 是否有回滚或撤销操作的机制 |
| 日志输出 | 是否记录关键步骤的输出和错误 |

### 运维适配审查

| 检查项 | 说明 |
|--------|------|
| 环境兼容 | 是否假定特定 OS/Shell/Python 版本 |
| 依赖声明 | 是否明确列出所需的外部命令和库 |
| 资源限制 | 是否会消耗过多 CPU/内存/磁盘 IO |
| 并发安全 | 多人同时执行是否会冲突 |

## 审查流程

### 步骤 1：自动检查

使用静态分析识别明显问题：
- 搜索硬编码凭证正则：`(password|secret|token|api_key)\s*=\s*['"]\S+['"]`
- 搜索危险命令：`rm -rf`、`DROP TABLE`、`shutdown`
- 检查是否有 `sudo` 或 `root` 提权

### 步骤 2：风险分级

根据检查结果给出风险评级：

| 问题数 | 最高严重度 | 建议 |
|--------|----------|------|
| 0 | - | ✅ 可以执行 |
| 1-2 | low/medium | ⚠️ 修正后执行 |
| 3+ | medium/high | 🔴 必须修正后重新审查 |
| 任意 | critical | 🚫 禁止执行，需重写 |

### 步骤 3：生成审查报告

```
📋 脚本审查报告

脚本：check_db_connections.sh
风险等级：⚠️ medium（1个中等风险 + 2个低风险）

✅ 通过项：
  • 无硬编码凭证
  • 超时控制已设置
  • 错误处理完善

⚠️ 需改进：
  1. [MEDIUM] 第15行：直接使用 root 执行，建议降权
  2. [LOW] 第22行：SQL查询缺少超时限制
  3. [LOW] 第30行：日志输出包含可能的敏感路径

📝 建议：修正上述问题后可执行。第1项如果确有需要需在审批中说明。
```

## 注意事项

- 审查结论不是"通过/不通过"二元判断，而是分级建议
- 低风险问题不阻塞执行，但应在下次迭代修正
- 所有高风险脚本入库前**必须**通过审查
