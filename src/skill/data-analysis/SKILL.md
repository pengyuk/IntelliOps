# 运维指标数据分析

> 改编自通用 Data Analysis Skill，针对运维场景定制。

## 何时使用

- 收到监控系统导出的指标数据（CSV/JSON）
- 需要对比故障前后的指标变化趋势
- 需要从原始指标数据中提取关键结论
- 需要验证根因假设的量化证据

## 支持的数据格式

### CSV 格式

```csv
timestamp,metric,value,unit,service
2026-06-01T10:00:00Z,cpu_usage,45.2,percent,payment-service
2026-06-01T10:01:00Z,cpu_usage,87.3,percent,payment-service
2026-06-01T10:00:00Z,db_connections,380,count,payment-service
2026-06-01T10:01:00Z,db_connections,450,count,payment-service
```

### JSON 格式

```json
[
  {"timestamp": "2026-06-01T10:00:00Z", "metric": "cpu_usage", "value": 45.2, "unit": "percent"},
  {"timestamp": "2026-06-01T10:01:00Z", "metric": "cpu_usage", "value": 87.3, "unit": "percent"}
]
```

## 分析流程

### 步骤 1：数据加载

识别用户提供的数据格式（CSV/JSON/表格），解析为结构化数据。

### 步骤 2：基础统计

对每个指标计算：
- 平均值、中位数、P95、P99
- 最大值、最小值
- 标准差（波动程度）

### 步骤 3：异常检测

识别以下异常模式：

| 模式 | 检测方法 | 运维含义 |
|------|---------|---------|
| 突增 | 当前值 > 基线均值 + 3×标准差 | 流量冲击或资源泄漏 |
| 趋势上升 | 连续 N 点持续上升 | 渐进式资源耗尽 |
| 周期性波动 | 固定时间间隔出现峰值 | 定时任务或批处理影响 |
| 断崖下跌 | 短时间内从正常值骤降至0 | 服务崩溃或采集中断 |

### 步骤 4：故障关联分析

如果提供了故障时间窗口，对比故障前后：

```
📊 指标对比分析 (故障窗口: 10:00-10:30)

指标             故障前均值    故障期间均值    变化
─────────────────────────────────────────────────
cpu_usage        35%          87%           +149%  ⚠️
db_connections   200          450           +125%  ⚠️
p99_latency      50ms         1850ms        +3600% 🔴
error_rate       0.1%         5.2%          +5100% 🔴
```

### 步骤 5：输出结论

```
📊 分析结论

1. p99_latency 从 50ms 飙升至 1850ms（+3600%），是最显著的异常指标
2. db_connections 同时期从 200 升至 450（+125%），与延迟高度相关
3. cpu_usage 升至 87%，可能是连接池压力导致的应用层 CPU 开销
4. 建议：优先排查数据库连接池瓶颈（连接数接近上限 500）
```

## 可视化建议

- 时间序列折线图：展示指标随时间变化
- 双轴图：db_connections（柱状）+ p99_latency（折线）展示相关性
- 热力图：按服务×指标展示异常程度

## 注意事项

- 确保数据已脱敏（去除 IP、主机名等）
- 样本量不足（<10个数据点）时统计结论可能不可靠
- 基线数据建议使用过去 7 天同时段均值
