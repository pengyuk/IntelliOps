# 架构设计（概要）

## 目标

为故障应急排查原型提供清晰的分层架构与模块边界，便于并行开发与后续扩展。

## 分层架构

- 数据层：多源采集、标准化、持久化（时序/关系/图）
- 知识层：本体（Ontology）+ 知识图谱（KG）+ 故障案例库
- 能力层：Skill 引擎、推荐/规则引擎、Harness 自动化执行
- 应用层：事故看板、现场排查助手、复盘管理、协同空间

## 关键设计原则

- 接口优先：明确数据与能力服务契约，前后端并行开发。
- 可观测与可审计：所有自动化/操作均记录日志并可回溯。
- 数据驱动：基于KG做推荐与关联，优先使用事实数据驱动决策。
- 渐进式演进：MVP->增强KG->增加自动化处置->闭环复盘。

## 依赖与技术选型建议

- 图数据库：Neo4j / JanusGraph（KG）
- 时序/监控：Prometheus / InfluxDB（指标）
- 关系/对象存储：Postgres + S3
- 后端服务：Python (FastAPI) 或 Node.js (NestJS)
- 前端：React + Ant Design / Vue + Element

## 交互与数据流（简述）

1. 告警/事件触发 → 数据层采集（T-OPM、优云导出）
2. 数据标准化并入KG，关联CMDB资源（T-CMDB）与变更记录
3. Skill/规则引擎生成排查建议并推送给应用层
4. Harness 执行模板化操作或生成工单，结果回写KG与工单系统
5. 复盘生成改进任务并同步到知识库与SOP
