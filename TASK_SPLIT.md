# 原型开发任务分工

## 总体思路

为提高原型开发效率，建议两人按“知识与数据能力 + 应用与协同能力”进行协作分工。

## 角色建议

### 1. 方案与数据架构负责人

主要负责：

- 数据层与知识层设计
- 本体建模与知识图谱框架
- 数据源接入与标准化
- 核心排查模型与推荐逻辑
- 复盘与知识沉淀机制

典型任务：

- 设计故障本体实体与关系
- 定义 T-CMDB、T-OPM、优云、工单、变更等数据模型
- 构建知识图谱实体关联逻辑
- 设计智能排查/根因推荐规则
- 输出数据接口与知识层规范

### 2. 应用与协同实现负责人

主要负责：

- 应用层原型落地
- 事故看板与现场排查助手
- 协同通讯与任务流转
- Skill 与 Harness 场景实现
- UI/交互与用户流程体验

典型任务：

- 设计事故看板与现场卡片界面
- 实现协同消息、角色通知、任务跟踪
- 编写应急操作模板与自动化流程
- 将 Skill 问答/推荐能力对接前端
- 设计复盘闭环与知识库更新体验

## 具体分工建议

### A. 方案与数据架构负责人

- 完成数据源分析与统一目录
- 设计 `ontology/` 文本或模型结构
- 定义知识图谱实体/关系视图
- 制定数据标准化规则与示例
- 输出后端服务接口规范
- 支持应用负责人对接知识能力

### B. 应用与协同实现负责人

- 完成原型界面框架与页面树
- 实现关键应用模块：事故看板、排查助手、复盘管理
- 设计并实现 Skill 交互场景
- 实现 Harness 自动化执行模板框架
- 完成协同机制与通知流程
- 负责前端/产品体验调优

## 协作方式

- 统一采用“接口优先、按阶段交付”
- 先同步数据模型与用例场景，再分头实现
- 每日/隔日对齐：数据结构、接口格式、关键流程
- 迭代原则：先做可用 MVP，再补全知识与自动化能力

## 推荐里程碑

1. 第一阶段：定义本体与数据模型；完成事故看板原型
2. 第二阶段：实现现场排查助手与知识图谱关联
3. 第三阶段：补充 Skill 建议、Harness 自动化、复盘闭环
4. 第四阶段：优化协同流程、增强角色通知与经验沉淀
