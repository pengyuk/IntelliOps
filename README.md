# 大模型驱动的应急认知原型研发

## 核心洞察

**项目定位**：新建原型研发（目标：大模型驱动的故障应急认知系统）

从"推荐下一步动作"转向"理解故障、推理根因、协同应急、自动学习"

## 概要

本项目面向信创场景，目标是构建一套真正**基于大模型认知能力**的故障应急系统。

原型旨在验证：
- 多源数据打通与知识化检索
- **大模型根因推理与故障认知**（核心差异）
- **协同智能体与知识自动蒸馏**（核心差异）
- 自动化处置与闭环学习

## 用户痛点与解决方案

| 角色 | 痛点 | 大模型解决 |
|------|------|----------|
| 运维/值班 | 知识沉淀不足，依赖个人经验，操作不可审计 | **推理链**弥补经验缺口 + **可信度**支撑决策 |
| 开发/DBA | 排查信息分散，沟通成本高，重复询问 | **War Room Agent** 同步状态 + **Investigation State Machine** 透明进度 |
| 管理层 | 复盘不闭环，知识无沉淀，新人培养慢 | **知识蒸馏**自动学习 + **演练系统**主动培养 |

## 原型目标（可量化）

**第一阶段**：MTTR 降低 15-20%  
**第二阶段**：MTTR 降低 30-40% | 诊断准确性 +40% | 推荐置信度 >70%  
**第三阶段**：复盘完成率 >=80% | 改进措施落实率 >=80% | 新人培养周期 -60%

## 核心能力与模块

### 核心能力矩阵

#### 平台基础能力
| 模块 | 功能 |
|------|------|
| 数据采集 | CMDB、告警、日志、变更、工单标准化入库 |
| 知识库 | 文档/案例库 + 向量检索，支持相似案例召回 |
| 事故管理 | 看板、事件流、状态机、Incident Graph |
| 执行层 | Action 执行、审计、回滚、通知 |

#### 大模型认知能力（核心差异化）
| 能力 | 作用 |
|------|------|
| **Root Cause Reasoner** | 从现象推导根因（多步因果链） |
| **Log Analyzer** | 日志自动摘要、异常检测、多源关联 |
| **Incident Copilot** | 多轮对话、动态诊断、证据收集 |
| **Credibility Framework** | 置信度评分、依据链、风险提示 |
| **Investigation State Machine** | 排查进度透明、状态追踪 |
| **Knowledge Distiller** | 从故障自动蒸馏规则、预警、SOP |
| **War Room Agent** | 多角色协同、自动时间线、智能推送 |
| **Incident Simulator** | 故障演练、新人评测、AI 反馈 |

详见设计方案：`UPGRADE_PLAN.md`。

## 原型三阶段交付路线

### 阶段 A：基础设施（周 1-2）
- ETL 管道 + 知识库搭建
- Incident Graph 初版
- 事故看板 + 基础 Action 执行
- **Person A**：准备 LLM API 环境、Prompt 框架  
- **Person B**：完成数据管道、UI 骨架

### 阶段 B：大模型认知（周 3-4）
- Root Cause Reasoner 上线
- Log Analyzer 集成
- Incident Copilot 上线
- Investigation State Machine
- 可信度框架集成
- **Person A**：推理链完整、评估验证  
- **Person B**：数据接口对接、结果展示

### 阶段 C：协同与闭环（周 5-7）
- War Room Agent 多角色协同
- Knowledge Distiller 自动学习
- Incident Simulator 演练系统
- 完整端到端验证
- **Person A**：知识蒸馏、规则提炼  
- **Person B**：演练框架、用户培养

## 关键要点

- **数据治理**：统一 ID、时间窗口、字段命名 → 为 LLM 提供一致数据源
- **接口约定**：Person A/B 在阶段 A 末必须完成接口 Schema 对齐
- **最小样本优先**：验证 1~2 个产品完整链路，而非全量上线
- **可信度设计**：所有推荐需要置信度、依据链、风险等级
- **审计与安全**：执行动作记录审计 ID、审批链路、结果跟踪

## 开发分工（两人·能力轴·低耦合）

### Person A：大模型认知引擎
**职责**：LLM 集成、推理链设计、Prompt 工程、能力评估

| 模块 | 交付 |
|------|------|
| Root Cause Reasoner | 推理链代码、Prompt 模板、推理结果接口 |
| Log Analyzer | 日志摘要逻辑、异常检测规则、输出格式 |
| Copilot 对话引擎 | 多轮对话状态机、问题生成、证据收集 |
| Credibility Framework | 置信度评分规则、可解释性输出 |
| Knowledge Distiller | 规则提炼逻辑、SOP 生成 |

**输入依赖**：Person B 提供标准化数据接口  
**输出契约**：JSON 格式的推理结果、置信度、依据链  
**里程碑**：LLM API 选型 → Prompt 开发 → 推理链验证

---

### Person B：数据平台与应用层
**职责**：ETL、知识库、Incident Graph、前端、执行、协同

| 模块 | 交付 |
|------|------|
| 数据采集与标准化 | CMDB、告警、日志、变更、工单 ETL 脚本 |
| 知识库 | 向量索引、相似案例检索、文档存储 |
| Incident Graph | 图存储、多跳查询、拓扑关系 |
| Investigation State Machine | 排查状态追踪、进度透明 |
| War Room Agent | 多角色通知、时间线生成、实时同步 |
| UI 与执行层 | 前端展示、Action 执行、审计记录 |
| Incident Simulator | 演练框架、评测接口 |

**输入依赖**：Person A 提供推理结果接口  
**输出契约**：标准化数据接口、推理结果展示、执行反馈  
**里程碑**：数据管道搭建 → 知识库验证 → UI 集成

---

## 大模型能力设计的 8 大缺口

详细分析见 `UPGRADE_PLAN.md`：

| # | 缺口 | 当前方案 | 目标形态 | 大模型价值 |
|---|---|---|---|---|
| 1 | 故障推理链 | 相似案例检索 | 多步因果推理 | 从"怎么做" → "为什么" |
| 2 | 日志认知 | 依赖人工查询 | 自动摘要+异常检测 | 从"海量日志" → "关键异常" |
| 3 | 动态问答 | 固定步骤推荐 | 多轮对话+状态记忆 | 从"流程驱动" → "交互驱动" |
| 4 | 协同智能体 | 人工通知 | War Room Agent 自动同步 | 从"被动等待" → "主动推送" |
| 5 | 状态跟踪 | 无记录 | Investigation State Machine | 从"重复询问" → "状态透明" |
| 6 | 知识蒸馏 | 人工复盘 | 自动提炼规则+SOP+脚本 | 从"文档沉淀" → "系统学习" |
| 7 | 经验传播 | 新人学习慢 | Incident Simulator 演练 | 从"被动积累" → "主动培养" |
| 8 | 可信度 | 黑盒推荐 | 置信度+依据链+风险提示 | 从"盲目信任" → "理性决策" |

## 关于知识层演进路线

采用"知识库优先"策略，逐步演进为大模型认知系统：

- **MVP**：知识库 + 向量检索（信息基础）
- **阶段 B**：+ Incident Graph + Root Cause Reasoner（大模型认知）
- **阶段 C**：+ 自动蒸馏 + 演练系统（闭环学习）

知识库不是最终形态，而是**大模型认知的基础数据源**。
