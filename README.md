# 故障应急排查原型

## 概要

本项目面向信创场景，目标是构建一套能把“人靠经验”的故障应急变成“流程+数据+工具”的可复用体系。原型旨在验证：多源数据打通、知识化检索、推荐式排查、自动化处置与复盘闭环的端到端能力。

## 背景与用户痛点

- 运维/值班：对信创栈场景知识缺乏系统沉淀，现场处置依赖个人经验，手工操作多且不可审计。
- 开发/DBA：排查时缺少可直接使用的上下文（变更、依赖、历史故障），沟通成本高。
- 管理层：复盘改进难以跟踪落实，无法量化改进效果。

## 项目目标（可量化）

- 在首批 1~2 个产品上把 MTTR（平均修复时间）降低 >= 30%。
- 实现复盘改进措施落实率 >= 80%（上线 3 个月内）。
- 自动化处置/模板化操作占比提升到 >= 40%（适用范围内）。

## 核心能力与模块

- 数据采集与标准化（T-CMDB、T-OPM、优云、变更、工单）
- 本体与知识图谱（实体/关系、相似历史检索）
- Skill 引擎（问答/推荐/步骤引导）
- Harness 自动化（脚本模板、审计、回滚）
- 应用层：事故看板、现场排查助手、复盘管理、协同通信

详见架构说明：`MODULE_ARCHITECTURE.md`。

## 可交付成果（MVP 范围）

1. 数据入图流水线（示例数据 + 简单 ETL 脚本）。
2. 基础知识图谱与查询 API（`/kg/query`）。
3. 事故看板与事件详情最小可用界面（`/ui`）。
4. 现场排查助手：推荐下一步排查动作（基于历史案例映射）。
5. 自动化 Action 模拟执行与审计接口（`/action/execute`）。

## 实施注意点

- 数据治理：统一实体 ID、时间窗口与字段命名，避免消歧不一致。
- 最小样本优先：先用 1~2 个产品的完整数据验证链路，再逐步泛化。
- 审计与安全：所有执行动作须记录审计 ID、执行者与审批链路。
- 权限与隐私：敏感日志/凭据控制访问与脱敏策略。

## 验收指标

- 事件响应时长（New -> Investigating）
- MTTR（平均修复时间）
- 复盘完成率与改进落实率
- 知识检索命中率（历史故障匹配）

## 开发分工建议（两人）

- `数据架构与知识层负责人`：负责 `src/ontology`、`src/kg`、ETL、数据接口与推荐规则。
- `应用与协同实现负责人`：负责 `src/ui`、`src/skill`、`src/harness`、前端/后端聚合 API 与复盘体验。

详见任务分工：`TASK_SPLIT.md`。

## 运行与快速验证

1. 安装依赖：

```bash
pip install -r requirements.txt
```

2. 启动后端（包含静态 UI）：

```bash
uvicorn src.backend.app:app --reload --host 0.0.0.0 --port 8000
```

3. 打开示例 UI： http://localhost:8000/ui/

## 路线与里程碑（8 周示例）

1. 周 1-2：本体初稿、样本数据准备、数据模型对齐。
2. 周 3-4：ETL + KG 入图、事故看板 MVP。
3. 周 5-6：现场排查助手、Action 模板库样例、简单 Skill 推荐。
4. 周 7-8：复盘链路完成、端到端验证、收集 KPI 基线。
